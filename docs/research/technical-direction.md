# 技術方針メモ

## 目的

このメモは、主手順書で実行する PoC の背景知識と判断材料を整理するための資料です。ここに書く内容は「即採用の決定」ではなく、採用候補と評価観点です。

## クライアントサイド実行方針

- 配信先が GitHub Pages のため、サーバーサイド処理は前提にしない
- 形状生成、プレビュー、エクスポートは可能な限りブラウザ内で完結させる
- 高負荷処理が発生する場合も、まずはクライアントサイド内での分離手段を検討する
- 外部 API 依存を前提にすると GitHub Pages 運用と整合しにくいため、PoC 段階では避ける

## OpenSCAD Wasm 採用候補

### 候補 A: OpenSCAD 本体を WebAssembly 化した実行基盤

- 長所: OpenSCAD 記法との互換性が最も高い可能性がある
- 短所: 初期ロードやバンドルサイズが重くなりやすい
- 向いている用途: 本番で OpenSCAD 構文互換性を重視したい場合

### 候補 B: OpenSCAD 互換を目指したブラウザ向けランタイム

- 長所: Web 統合しやすい可能性がある
- 短所: OpenSCAD の全機能をそのまま使えない可能性がある
- 向いている用途: まずブラウザ PoC を通して UX を確認したい場合

## 評価観点

- `difference()`, `union()`, `linear_extrude()` など基本形状演算が安定して動くか
- 将来的に `text()` とフォント依存を扱える見込みがあるか
- ブラウザでの初期化コストが許容範囲か
- エラー内容を UI 側で扱いやすいか
- STL / 3MF への出力導線を持てるか

## 3MF と STL の扱いの違い

- STL は単一メッシュ交換に向いており、まず成立確認しやすい
- 3MF はメタデータや複数構成要素を持てる可能性があり、表現力は高い
- ただし 3MF は「生成できること」と「スライサーで期待どおり読めること」が別問題になりやすい
- そのため、最初に STL を押さえ、その後 3MF を検証する順が安全

## 色付き 3MF の不確実性

- ブラウザ内での生成経路が安定していない可能性がある
- スライサー側の解釈差異で、色情報や複数要素の扱いが変わる可能性がある
- アプリ内で色指定できても、最終出力で同じ意味を持つとは限らない
- そのため、色付き 3MF を前提に本番設計を固定するのは早い

## 別体積方式を有力候補とする理由

- 本体と印字を別ボディで持てば、色指定の有無に依存しにくい
- スライサー側で部品単位に扱えるため、運用上の説明がしやすい
- 3MF が不安定でも、複数 STL や複数ボディ出力に退避しやすい
- プレビューと export で責務を分けやすく、実装の見通しも立てやすい
