# 出力形式メモ

## STL の長所

- 対応ツールが多く、PoC の成立確認に向いている
- 単純なメッシュ交換として扱いやすい
- 最低限の出力経路として確保しやすい

## STL の短所

- 色、材質、複数パーツ関係の表現力が低い
- 単一メッシュ前提になりやすく、複数要素の意味を持たせにくい
- アプリ側で豊富な情報を持っていても、出力時に落ちやすい

## 3MF の長所

- 複数構成要素や付加情報を持てる余地がある
- うまく成立すれば、将来的な出力表現の幅が広い
- 単なるメッシュ交換よりもリッチなデータを扱える可能性がある

## 3MF の短所

- ブラウザ内で安定生成できるかが未確定
- 生成できても、スライサー側で期待どおり読まれるとは限らない
- 色付き 3MF はさらに相互運用の不確実性が高い

## Bambu Studio 連携時の考慮点

- 読み込み時の単位がずれないか
- 複数ボディが別要素として認識されるか
- 色やパーツ情報が保持されるか
- 本体と印字の位置関係が崩れないか
- 実運用で毎回説明なしに使える安定性があるか

## 色だけに依存する設計のリスク

- 出力形式や読み込み先で色情報の意味が変わる可能性がある
- 色分けできても、物理的に別パーツとして扱えない場合がある
- 運用上の説明責任が「このスライサーではこう見える」に引きずられやすい
- 実装側の UI と最終製造物の対応が不安定になりやすい

## 本体 / 印字を別ボディで持つ設計のメリット

- 色がなくても部位の違いを保持できる
- 複数 STL や多ボディ 3MF に退避しやすい
- スライサー上での調整余地を残せる
- 将来、インレイ、別素材、交換式パーツへ拡張しやすい

## 実務上の結論

- 最低限の成立確認は STL から始める
- 3MF は有望でも、PoC で安定性を確認するまでは正式採用しない
- 本番設計は色指定だけに依存させず、body / legend 分離可能な構造で進める

## 現時点の PoC 実装メモ

- Task 03 では OpenSCAD WASM から binary STL を直接出力する経路を実装した
- Task 04 では OpenSCAD WASM から OFF を出力し、ブラウザ側で 3MF パッケージを組み立てる経路を実装した
- この 3MF 経路は「3MF をブラウザ内で成立させる」ための PoC であり、Bambu Studio 互換性は Task 05 で別途確認する
